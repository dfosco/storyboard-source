❯ Am I reinventing OpenAPI / MCP in a shitty way?


● Short answer: partially yes, but not entirely.

  What you're reinventing (and shouldn't):

   - Tool/API schemas — the canvas HTTP endpoints + CLI flags + widget type/props shapes are exactly what OpenAPI
  describes. You're hand-rolling AGENTS.md prose where a spec would auto-generate docs, validate inputs, and give
  agents a discoverable surface.
   - Tool catalogs for LLMs — MCP exists precisely so agents can ask "what tools/resources are available?" and get
  typed answers. Your cat .storyboard/terminals/*.json + "read AGENTS.md" pattern is a bespoke version of MCP's
  tools/list + resources/list.
   - Discriminated-union schemas for widgets — type: "sticky-note" | "markdown" | "prototype" | ... with per-type
  props is a textbook JSON Schema oneOf + discriminator. Right now it lives only in renderer code.

  What is not reinventing those, and is actually yours:

   - Spatial semantics (--near, --direction, collision resolution, gap units) — not in OpenAPI/MCP. This is
  canvas-domain.
   - The agent-as-widget-on-canvas model, messaging, hubs, broadcast, terminal buffers — also genuinely novel; MCP
  has no concept of "agents sit at coordinates and talk through connectors."
   - The HMR/WebSocket-driven live canvas — orthogonal to API specs.

  Pragmatic path:

   1. Use MCP for the canvas API. Wrap the existing HTTP endpoints as an MCP server. Agents get typed tool
  discovery for free; you stop writing prose schemas.
