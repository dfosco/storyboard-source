# Fix Canvas Widget Persistence

## Goals

1. Widget deletions must persist reliably — deleted widgets never reappear after page reload
2. Widget moves and property changes must not revert — no stale `widgets_replaced` can overwrite fresh changes
3. All write operations (add, update, move, delete) must be serialized to prevent race conditions

## Problem Statement

The canvas persistence pipeline has a race condition between `debouncedSave` and `queueWrite`:

- **`handleWidgetUpdate`** triggers a 2-second debounced save that sends the FULL widget array via `PUT /update` → `widgets_replaced` event. This debounced save bypasses `queueWrite`.
- **`handleWidgetRemove`** immediately queues a `DELETE` → `widget_removed` event through `queueWrite`.
- **Race**: Edit widget → delete widget within 2 seconds. Delete fires immediately. Then debounced save fires with stale array (still includes deleted widget) → `widgets_replaced` restores it.

Same race affects moves: drag a widget (triggers debounced save with old positions), then drag again quickly. The first debounced save can overwrite the second position.

## Approach

Route ALL writes through the `queueWrite` serializer. Cancel pending debounced saves before delete operations. Ensure `debouncedSave` always uses the latest widget state.

## Files to Change

- `packages/react/src/canvas/CanvasPage.jsx` — fix debouncedSave to go through queueWrite, cancel on delete

## Steps

1. In `handleWidgetRemove`: cancel `debouncedSave` before queuing the delete
2. In `handleWidgetUpdate`: route `debouncedSave` through `queueWrite` so it's serialized with deletes
3. In `handleItemDragEnd` (single-widget drag): same — cancel debounced save, use queueWrite for the immediate save
4. Verify the `debouncedSave` callback reads fresh state (not stale closure)
5. Run tests, lint, build

## Edge Cases & Risks

- The debounced save must capture the LATEST widget state at fire time, not the state when it was scheduled
- Cancelling debounced save on delete is critical — otherwise deleted widgets reappear
- `queueWrite` must handle async debounce (the debounce fires, but the actual API call goes through the queue)
