# Canvas Unique ID Migration

## Goals

1. **Every consumer of canvas identity uses the canonical path-based ID** (e.g., `design/overview` instead of bare `overview`) so that canvases with the same basename in different folders never collide.
2. **Consistent variable naming** — rename `canvasName`/`name` to `canvasId` everywhere it represents the canvas identity (not the human-readable title), making the codebase self-documenting.
3. **Remove legacy basename fallbacks** — eliminate `findCanvasPath()` basename fallback and fix HMR events that use `parsed.name` instead of `parsed.id`, preventing silent failures for prototype-scoped canvases.

## Problem Statement

The canvas system already has path-based canonical IDs (`toCanvasId()` in `identity.js`) that produce unique composite IDs like `design/overview` or `proto:Main/board`. However, many consumers inconsistently use bare basenames or different variable names (`name`, `canvasName`, `canvasId`) to refer to the same concept. This creates:

- **Collision risk**: Two canvases in different folders with the same filename would conflict in HMR tracking, viewport persistence, and copy/paste identity.
- **Prototype canvas HMR bug**: HMR events send `parsed.name` (e.g., `Main/board`) but consumers compare against `parsed.id` (e.g., `proto:Main/board`), causing silent missed updates.
- **WidgetChrome bug**: "Copy widget ID" extracts canvas identity from URL path's last segment, losing folder context.
- **Rename-watcher bug**: `canvasRoute()` strips `.folder` segments entirely instead of keeping the folder name, producing wrong routes for folder-based canvases.

## Approach

Standardize on `canvasId` as the variable name for the canonical path-based identity everywhere. Use `toCanvasId()` (already canonical) as the single source of truth. Propagate through the full stack: data-plugin → context → CanvasPage → bridge state → CoreUIBar → API calls → server.

## Naming Convention

- **`canvasId`** — the canonical path-based identity (e.g., `design/overview`, `proto:Main/board`). Used for indexing, API calls, bridge state, HMR events, persistence keys, copy/paste.
- **`canvasTitle`** — human-readable display name (e.g., "Design Overview"). Keep as-is where used for UI display.
- Keep the `name` field in API payloads for now to avoid server API breaking changes — document as alias for `canvasId`.

## Edge Cases & Risks

- **Bridge state key change** (`name` → `canvasId`) affects cross-framework boundary (React ↔ Svelte). Must update both sides atomically.
- **Server API backward compat** — CLI commands (`canvasAdd`, `canvasRead`, `canvasRead`) send `name` in payloads. Keep server accepting `name` and internally alias to `canvasId`.
- **Viewport persistence** — `sb-canvas-viewport:${canvasId}` keys will change for canvases that were previously stored under basename. Users will lose saved viewport state for affected canvases (one-time reset, acceptable).
- **Clipboard format** — `canvasId::widgetId` copy/paste strings remain compatible since CanvasPage already uses the canonical ID.
- **Production builds** — `toCanvasId()` is used at build time by data-plugin. No runtime impact.
