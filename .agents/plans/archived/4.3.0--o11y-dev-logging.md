# O11y Dev Logging — Post-Release Plan

## What shipped

Two commits on `4.3.0--prompt-widget` added structured observability logging to the storyboard dev server:

1. **`f9a7c622`** — Core JSONL logger (`devLogger.js`) + instrumented all API error responses
2. **`96ab0af0`** — Console cleanup + browser error bridge via HMR

### Current state

- **Server errors**: Every `sendJson(res, 4xx/5xx, ...)` across both server layers (Vite middleware + standalone server) writes a JSONL entry to `.storyboard/logs/{date}.jsonl` with route, method, URL, error message, devDomain, branch, and auto-extracted context (canvas name, prototype, page).
- **Server warnings**: `console.warn/error` in canvas/server.js, hot-pool.js, terminal-server.js, and selectedWidgets.js replaced with `devLog().logEvent()`.
- **Browser errors**: Injected client-side script patches `console.error`, `console.warn`, `window.onerror`, and `unhandledrejection` → forwarded to server via `storyboard:client-error` HMR event → written to `.storyboard/logs/` with `source: "browser"`, route, stack, line/col.
- **Retention**: 7-day auto-pruning on startup.
- **Console policy**: Logs always write to disk. `featureFlags.dev-logs` controls verbose console output only.

## Post-Release Improvements

### Phase 1 — Validation & quick wins

#### 1.1 — Live validation
Run the dev server, trigger real errors (404 canvas, bad widget ID, broken prototype route), and verify the log entries capture the right context. Check browser errors appear with useful stack traces.

#### 1.2 — Add `storyboard logs` CLI command
Add a CLI command to read/tail/filter logs from `.storyboard/logs/`:

```
storyboard logs                  # tail today's log
storyboard logs --level error    # filter by level
storyboard logs --since 2h       # last 2 hours
storyboard logs --route canvas   # filter by route
storyboard logs --source browser # browser errors only
storyboard logs --json           # raw JSONL output (for piping to jq)
```

This makes logs accessible without remembering the file path or jq syntax.

#### 1.3 — Agent-readable log summary API
Add a `GET /_storyboard/logs` API endpoint that returns recent errors as JSON. This lets canvas agents read errors programmatically without filesystem access:

```json
GET /_storyboard/logs?since=1h&level=error
{
  "entries": [...],
  "summary": { "errors": 3, "warnings": 12, "period": "1h" }
}
```

### Phase 2 — Deeper instrumentation

#### 2.1 — Vite transform/build errors
Capture Vite's own error events (module transform failures, dependency resolution errors) and pipe them through `devLogger.logEvent()`. These currently only show in the terminal.

#### 2.2 — Data plugin errors
Instrument `data-plugin.js` to log when flow/object/record files fail to parse, have duplicate names, or contain unresolvable `$ref`s. Currently these are `console.warn` in the Vite plugin.

#### 2.3 — WebSocket / terminal errors
Instrument terminal-server.js WebSocket connection errors, PTY spawn failures, and tmux session conflicts with structured log entries including widget ID, canvas ID, and session name.

#### 2.4 — HMR event logging
Log significant HMR events (full reloads, canvas file changes, config changes) as `info` level entries. Useful for correlating "the page went blank" with "a full reload was triggered by X".

### Phase 3 — Observability UX

#### 3.1 — Canvas DevTools log viewer
Add a log viewer panel to the canvas DevTools (behind `dev-logs` feature flag). Shows a scrollable, filterable list of recent errors with expandable stack traces. Real-time updates via HMR events.

#### 3.2 — Error toast notifications
When a server or browser error is logged, show a non-intrusive toast in the canvas UI (behind `dev-logs` flag) with the error message. Clicking the toast opens the DevTools log viewer filtered to that error.

#### 3.3 — Error correlation
Add a `requestId` to each middleware request (UUID stamped on `res.__sbLogCtx`). Browser errors could include the current page/canvas context. This enables correlating "the canvas CRUD failed" with "the browser showed a blank widget" across the same user action.

### Phase 4 — Operational

#### 4.1 — Log size monitoring
Add a health check that warns when daily log files exceed a threshold (e.g. 1MB). High error rates could indicate a real bug vs. normal usage.

#### 4.2 — Configurable retention
Move the 7-day retention period to `storyboard.config.json` so projects with different needs can adjust it.

#### 4.3 — Log export for bug reports
Add `storyboard logs export` CLI command that bundles recent logs into a compressed archive for attaching to bug reports or GitHub Issues.

## Notes

- Client-side stores (`toolRegistry`, `configStore`, `modes`, `commandActions`) still use `console.error/warn` — these are browser-side singletons and now captured by the browser error bridge. No changes needed.
- CLI tools (`storyboard agent`, `storyboard terminal send`, etc.) still use `console.error` — these are user-facing terminal output, not server logs. Leave as-is.
- The `cli/proxy.js` warnings during Caddy setup are CLI-time output, not dev-server runtime. Leave as-is.
