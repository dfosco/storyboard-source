# Prompt Widget â€” Implementation Plan

## Goals

1. **Users can type a prompt in a canvas widget and have an AI agent process it in the background**, with visual feedback showing pending/done/error states.
2. **Connected widgets provide context to the prompt** â€” e.g. connecting a GH issue embed lets the user say "Summarize this issue" and the agent receives the connected widget data.
3. **Every prompt execution produces a visible output widget on the canvas** â€” the agent always creates a result widget (markdown, component, etc.) positioned next to the prompt widget.

## Problem Statement

The canvas currently has no way to invoke AI agents. Users need a widget that accepts natural language prompts, runs them through a background agent session, and renders results directly on the canvas. The widget must handle the full lifecycle: input â†’ processing â†’ output/error.

## Approach

### Architecture: 3-layer design

1. **PromptWidget.jsx** (React) â€” UI with text input, submit button, status indicator (idle/spinning/done/error). Calls the prompt server API on submit.
2. **prompt-server.js** (Core, Node) â€” Server-side endpoint mounted on the canvas Vite plugin. On submit:
   - Spawns a background `copilot` CLI process with the prompt
   - Injects a system prompt that tells the agent to: output results as a canvas widget via the canvas API, signal done/error via a completion file
   - Monitors completion and streams status back via a polling endpoint
3. **Widget connections** â€” When the prompt widget has connected widgets (new concept: `connections` array in widget props), their content is serialized and injected into the agent's context.

### Key Design Decisions

- **Agent execution**: Use `copilot` CLI in non-interactive mode. The agent runs in the repo working directory so it has full file access.
- **Completion signaling**: Agent writes a JSON status file to `.storyboard/.prompt-sessions/<sessionId>.json` with `{ status: 'done'|'error', resultWidgetId?, error? }`. The prompt server polls this.
- **Output widget creation**: The system prompt instructs the agent to call the canvas API (`POST /_storyboard/canvas/widget`) to add result widgets. The prompt widget stores the result widget ID for visual linking.
- **Connections**: Stored as `props.connections: string[]` (array of widget IDs). The prompt server reads connected widget data from the canvas state and serializes it into the agent context.
- **No `useState` in the widget** for persistent state â€” widget status (`idle`, `pending`, `done`, `error`) is stored in `props.status` via `onUpdate`. Session ID in `props.sessionId`.

## Implementation Steps

### Step 1: Widget config & registration
- Add `prompt` type to `widgets.config.json` with props: `text`, `status`, `sessionId`, `resultWidgetId`, `errorMessage`, `connections`, `width`
- Register `PromptWidget` in `widgets/index.js`
- Export `promptSchema` from `widgetProps.js`

### Step 2: Prompt server
- Create `prompt-server.js` with endpoints:
  - `POST /_storyboard/prompt/execute` â€” Accepts `{ canvasName, widgetId, prompt, connections, connectedWidgets }`, spawns agent, returns `{ sessionId }`
  - `GET /_storyboard/prompt/status/:sessionId` â€” Returns `{ status, resultWidgetId?, error? }`
- Agent spawning: runs `copilot` CLI with injected system prompt
- System prompt template includes:
  - Connected widget context (serialized)
  - Instructions to add result widget via canvas API
  - Instructions to write completion signal file
  - Error handling instructions

### Step 3: PromptWidget component
- Input state: Text input + submit button (uses local `useState` only for the input text before submission â€” all persistent state in props)
- Status rendering:
  - `idle` (default): Show input form
  - `pending`: Show spinner icon, disable input
  - `done`: Show checkmark, link to result widget
  - `error`: Show error icon, click to open terminal session
- On submit: POST to prompt server, receive sessionId, call `onUpdate({ status: 'pending', sessionId, text })`, start polling
- Polling: `setInterval` checking status endpoint, on done/error calls `onUpdate({ status, resultWidgetId|errorMessage })`

### Step 4: Widget connections
- Add connection UI: When prompt widget is selected, show connection handles
- Store connections in `props.connections` (array of widget IDs)
- On prompt execution, resolve connected widget data from local canvas state
- Serialize connected widget content for agent context

### Step 5: Output widget creation
- System prompt instructs agent to determine output type based on user request:
  - Text/summary â†’ markdown widget
  - Code/component â†’ component or code markdown widget
  - Default â†’ markdown widget
- Agent calls canvas API to create output widget positioned to the right of the prompt widget
- Widget position: `promptWidget.position.x + promptWidget.width + 40, promptWidget.position.y`

## Edge Cases & Risks

- **Agent not installed**: `copilot` CLI may not be available â€” detect and show helpful error
- **Agent hangs**: Set a timeout (5 minutes) â€” if no completion signal, mark as error
- **Canvas API unavailable**: Agent runs in same Vite server context so canvas API should always be available
- **Stale polling**: Clear interval on unmount, on widget removal, and when status is terminal
- **Multiple prompts**: Each execution gets a unique sessionId â€” concurrent prompts are independent
- **HMR**: Widget must survive HMR â€” persist all state in props, reconnect polling on mount if status is `pending`
- **Connected widget deleted**: Gracefully handle missing connections â€” filter out stale IDs before execution
