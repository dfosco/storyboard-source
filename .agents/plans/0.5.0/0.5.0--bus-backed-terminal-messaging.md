# Phase 4: Bus-Backed Terminal Messaging + Presence Registry

## Problem

The current terminal messaging system is point-to-point:
- `storyboard terminal send` → HTTP → tmux send-keys (if live) or pendingMessages JSON (if not)
- No shared log, no queryable history, no presence awareness
- Messages are lost if the JSON config is cleared or corrupted

## Approach

Replace with bus-backed messaging where:
1. All messages flow through the JSONL event bus (source of truth)
2. A server-side delivery bridge subscribes in-process and injects into tmux
3. A presence registry tracks agent availability with heartbeats
4. CLI `terminal send` publishes to the bus and waits for delivery ack
5. Durable cursors ensure at-least-once delivery across restarts

## Architecture

```
CLI (terminal send)
  → POST /_storyboard/messages/publish
    → bus.publish(channel, envelope)
      → JSONL append
      → in-process subscribers notified
        → DeliveryBridge: tmux send-keys + ack
        → WebSocket push to browser
```

### Channel Naming

Scoped channels to avoid cross-branch/canvas collisions:
- Terminal inbox: `terminal:{branch}:{canvasId}:{widgetId}`
- Presence: `presence:{branch}:{canvasId}`

### Durable Cursors

Each terminal has a `lastDeliveredEventId` persisted in `.storyboard/messages/cursors/{widgetId}.json`.
On bridge start: `read(channel, { since: cursor })` → deliver backlog → go live.
After successful tmux inject: advance cursor + emit `message:delivered`.

## Todos

### Phase 4A: Presence Registry
- `presence.js` — PresenceRegistry class: join/heartbeat, in-memory Map rebuilt from bus log on startup
- Presence event types: `presence:join`, `presence:heartbeat` (no explicit deregister — agents expire naturally)
- HTTP endpoints: `GET /presence/:canvasId`, `GET /presence/:canvasId/:widgetId`
- Auto-expire entries after 90s without heartbeat (3 missed beats = gone)
- Integration: register on agent startup, 30s heartbeat interval, no deregister (crash-safe by design)

### Phase 4B: Delivery Bridge
- `delivery.js` — DeliveryBridge class: subscribes to terminal channels in-process (NOT SSE)
- Durable cursors: read `lastDeliveredEventId` from disk, backfill missed messages, then go live
- tmux injection: format message + send-keys into tmux session
- On success: advance cursor, emit `message:delivered` on bus
- On failure: emit `message:failed`, leave cursor unadvanced (retry on next bridge start)
- Rebind on widget identity assignment (hot-pool reassignment), not just process start/stop
- Lifecycle: `bindWidget(widgetId, tmuxSession)` / `unbindWidget(widgetId)`

### Phase 4C: Rewrite terminal/send
- Rewrite `server.js:2629` terminal/send handler → publish to bus channel
- Rewrite `terminal-messaging.js:handleSend()` → POST to `/_storyboard/messages/publish`
- CLI waits for correlated `message:delivered` or `message:failed` response (timeout 5s)
- Include sender metadata (widgetId, displayName, canvasId, branch)

### Phase 4D: Startup Migration & Cleanup
- One-time migrator: read existing `pendingMessages` from terminal configs → publish to bus → clear config
- Replace `deliverPendingMessages()` in terminal-server.js with delivery bridge subscription
- Wire bridge into terminal-server.js session lifecycle (start/stop/reassign)
- Remove old pendingMessages code paths after migration is stable

### Phase 4E: Tests
- Presence registry: join/heartbeat/expiry/replay (no deregister)
- Delivery bridge: cursor persistence, backfill, tmux injection mock, rebind
- CLI send: publish + wait for ack, timeout handling
- Integration: end-to-end send → deliver → ack

## Key Decisions

1. **Crash-safe presence** — no explicit deregister; 30s heartbeat, 90s expiry (3 missed = gone)
2. **In-process bridge** — uses `bus.subscribe()` directly, not HTTP SSE (same process)
3. **Presence is per-canvas** — all agents on a canvas share one presence channel
4. **One-time migration only** — no long-term pendingMessages fallback
5. **Hot-pool aware** — bridge rebinds on widget identity change, not just process lifecycle
