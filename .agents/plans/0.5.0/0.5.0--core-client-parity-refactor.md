# Plan: Core ‚Üî Client Parity Refactor

## Problem

The storyboard-core monorepo uses **30+ Vite aliases** in `vite.config.js` that redirect `@dfosco/storyboard-*` package imports to local source files. This creates a fundamental divergence: the core repo resolves imports through source paths with HMR, while npm consumers resolve through `package.json` exports and pre-built bundles.

This divergence has repeatedly caused bugs that "work in core, break in client":
- `React.createElement` not defined (JSX transform differs: source uses plugin, dist uses esbuild classic)
- React version mismatch (react-dom bundled in dist but externalized differently in source)
- Leaked monorepo-relative imports (`../../../core/`) that pass in core but fail in consumers
- Missing dependencies (used transitively in core but undeclared in package.json)

## Goal

Make the core repo's `src/` example app behave **as close as possible** to a real npm consumer ‚Äî same resolution paths, same bundle artifacts, same dependency tree ‚Äî while preserving fast dev iteration (HMR, source maps).

## Current State (30+ aliases)

```
vite.config.js resolve.alias:
  @dfosco/storyboard-core          ‚Üí packages/core/src/index.js
  @dfosco/storyboard-core/ui-runtime ‚Üí packages/core/src/ui-entry.js  ‚Üê biggest divergence
  @dfosco/storyboard-core/ui-runtime/style.css ‚Üí packages/core/src/styles/tailwind.css
  @dfosco/storyboard-core/config   ‚Üí packages/core/src/configSchema.js
  @dfosco/storyboard-core/comments ‚Üí packages/core/src/comments/index.js
  @dfosco/storyboard-core/vite/*   ‚Üí packages/core/src/vite/*
  @dfosco/storyboard-core/canvas/* ‚Üí packages/core/src/canvas/*
  ... (25+ more subpath aliases)
  @dfosco/storyboard-react         ‚Üí packages/react/src/index.js
  @dfosco/storyboard-react/vite    ‚Üí packages/react/src/vite/data-plugin.js
  @dfosco/storyboard-react-primer  ‚Üí packages/react-primer/src/index.js
  @dfosco/storyboard-react-reshaped ‚Üí packages/react-reshaped/src/index.js
  @dfosco/tiny-canvas              ‚Üí packages/tiny-canvas/src/index.js
```

`vitest.config.js` mirrors the `ui-runtime` aliases.

## Approach

### Phase 1: Eliminate source-serving aliases via npm workspaces

**Replace all aliases with npm workspace resolution.** npm workspaces already resolve `@dfosco/storyboard-core` to `packages/core/` ‚Äî but the `package.json` exports map controls which files are exposed. The current exports map already covers every subpath the aliases map to.

The key insight: Vite resolves workspace packages through their `package.json` exports, which point to `src/` files (not `dist/`). So `import { init } from '@dfosco/storyboard-core'` already resolves to `packages/core/src/index.js` ‚Äî the alias is redundant.

**Steps:**

1. **Audit which aliases are actually needed** ‚Äî for each alias, check if the package.json exports map already provides the same resolution. Remove redundant aliases.

2. **Add missing exports to package.json** ‚Äî if any alias maps to a path not in the exports map, add it. This benefits both the monorepo and consumers.

3. **Remove all redundant aliases from vite.config.js** ‚Äî keep only genuinely needed ones (likely just `@` ‚Üí `./src` for the example app's own source).

4. **Test that the example app still works** ‚Äî run `storyboard dev` and verify all features work without aliases.

### Phase 2: Fix the ui-runtime divergence

This is the biggest gap. In core, `@dfosco/storyboard-core/ui-runtime` resolves to **source** (`ui-entry.js`) which Vite serves as individual modules with HMR. In consumers, it resolves to the **pre-built bundle** (`dist/storyboard-ui.js`).

**Option A: Always use the pre-built bundle (even in dev)**

- Remove the `ui-runtime` alias
- Run `npm run build:ui -- --watch` in parallel with `storyboard dev`
- Vite serves `dist/storyboard-ui.js` which auto-rebuilds on source changes
- Pro: Exact parity with consumers
- Con: Slower iteration (rebuild on every change), no HMR for toolbar components

**Option B: Make the export map point to source when in the monorepo**

- Use conditional exports: `"./ui-runtime": { "development": "./src/ui-entry.js", "default": "./dist/storyboard-ui.js" }`
- Vite in dev mode picks the `development` condition
- Published package uses `default` ‚Üí dist
- Pro: HMR preserved, no alias needed
- Con: Conditional exports are a Node.js/bundler feature ‚Äî may not work identically everywhere

**Option C (Recommended): Keep one alias, just for ui-runtime**

- Remove all other aliases (Phase 1 handles those)
- Keep only `@dfosco/storyboard-core/ui-runtime` ‚Üí source alias
- Document it as the single intentional divergence
- Add a CI smoke test that catches issues this creates (see Phase 4)
- Pro: Pragmatic, HMR preserved, minimal divergence
- Con: One remaining alias ‚Äî but it's well-documented and tested

### Phase 3: Add missing dependencies & fix build config

Based on the bugs found during this session:

1. **Declare all runtime deps in package.json** ‚Äî run `depcheck` or equivalent on each package to find undeclared imports. Known missing: `cmdk`, `@radix-ui/react-dialog`, `@radix-ui/react-visually-hidden`, `feather-icons` (already fixed in beta.19).

2. **Fix vite.ui.config.js** ‚Äî already done in beta.22:
   - Remove Svelte plugin (Svelte was removed)
   - Add `esbuild.jsx: 'automatic'`
   - Use regex externals for react/react-dom subpaths

3. **Add `react` and `react-dom` as peerDependencies of storyboard-core** ‚Äî the ui-runtime bundle imports from them but they're not declared anywhere.

### Phase 4: Add consumer smoke tests

Add a CI step that verifies the published packages work correctly:

1. **Pack test** ‚Äî `npm pack` each package, install the tarballs in a fresh temp project, and verify basic imports resolve:
   ```bash
   npm pack packages/core packages/react
   mkdir /tmp/smoke && cd /tmp/smoke
   npm init -y && npm install ../../*.tgz react react-dom
   node -e "require('@dfosco/storyboard-core')"
   node -e "require('@dfosco/storyboard-react')"
   ```

2. **Bundle test** ‚Äî create a minimal Vite project that imports `mountStoryboardCore` and `StoryboardProvider`, runs `vite build`, and verifies no import errors.

3. **ui-runtime import test** ‚Äî specifically test that `import('@dfosco/storyboard-core/ui-runtime')` resolves and exports `mountDevTools`.

### Phase 5: Align vitest.config.js

Remove the `ui-runtime` aliases from vitest.config.js. Tests that import from `@dfosco/storyboard-core/ui-runtime` should resolve through the same path as consumers ‚Äî either the package export or (if Option C) a minimal alias that matches the one in vite.config.js.

## Risks & Considerations

- **HMR for toolbar development** ‚Äî removing the ui-runtime alias means toolbar changes require a rebuild. This is a real DX tradeoff. Option C mitigates this.
- **Workspace resolution edge cases** ‚Äî npm workspaces + package.json exports can behave differently from direct npm installs in subtle ways. The smoke test (Phase 4) catches these.
- **Breaking changes for existing clients** ‚Äî all changes should be backward-compatible. New exports in package.json don't break existing consumers.
- **Svelte removal cleanup** ‚Äî several aliases reference `svelte-plugin-ui/` paths. These should be removed or renamed as part of the Svelte ‚Üí React migration cleanup.

## Order of Operations

1. Phase 1 (alias elimination) ‚Äî lowest risk, biggest simplification
2. Phase 3 (dependency fixes) ‚Äî already partially done, finish it
3. Phase 4 (smoke tests) ‚Äî prevents future regressions
4. Phase 2 (ui-runtime) ‚Äî biggest impact, needs the most care
5. Phase 5 (vitest alignment) ‚Äî follow-up after Phase 2

## Notes

- All changes should happen on the `4.2.0` branch as patches
- Each phase can be released as a separate beta
- The client repo (`storyboard/.worktrees/unified-vision`) serves as the real-world integration test
