# WebGL Context Virtualization for storyboard-core

## Problem

`storyboard-core` uses `ghostty-web` (Zig → WASM) for canvas terminal widgets, and each terminal widget currently creates its own WebGL context with no shared manager or awareness of browser limits.

Browsers typically allow only ~8-16 WebGL contexts per page. Once that limit is exceeded, new terminals may fail to initialize, older contexts may be evicted, rendering can go blank, and interaction becomes unreliable.

This matters on multi-terminal canvases because a single storyboard can contain many terminal-like widgets at once:

- agent terminals
- prompt widgets
- bare shell terminals
- hot-pool prewarmed sessions (up to 3 each for terminal, prompt, copilot, claude, codex)

The result is that large canvases can hit the GPU/context ceiling long before they hit layout or PTY limits.

## Approach

Use **viewport-based context virtualization**.

Only terminals that are visible, focused, or otherwise active should hold a live WebGL context. Offscreen terminals should release their live renderer and fall back to a frozen snapshot image of their most recent terminal state.

This keeps active terminals at full `ghostty-web` performance while treating offscreen terminals more like virtualized list rows: preserved logically, but not fully rendered.

## Implementation Sketch

### 1. Visibility tracking

Add an `IntersectionObserver` to terminal widget containers in `packages/react/src/canvas/widgets/TerminalWidget.jsx`.

Each widget reports state such as:

- visible in viewport
- near viewport (optional preload margin)
- focused / selected
- expanded / actively streaming

### 2. Context pool manager

Introduce a page-level WebGL context manager for terminal widgets.

Responsibilities:

- cap concurrent live WebGL terminals below the browser limit
- prioritize visible + focused terminals first
- reserve a small safety buffer for new activations
- hand out / reclaim live renderer slots as widgets scroll in and out

Conceptually:

- **live mode**: widget owns a `ghostty-web` renderer + active WebGL context
- **frozen mode**: widget keeps PTY/session identity but shows a snapshot instead of a live canvas

### 3. Snapshot fallback

Reuse the existing rolling terminal snapshot infrastructure in `terminal-server.js`:

- 5-minute private rolling buffer
- 1-minute public snapshots
- ANSI-stripped output already available for derived rendering

For offscreen widgets, render a frozen terminal preview from the latest buffered snapshot. This can start as a styled text/ANSI-derived image layer and later evolve into a more pixel-faithful bitmap capture if needed.

### 4. Activation / deactivation flow

- when a terminal enters the viewport, request a live slot from the pool
- if granted, hydrate / reattach `ghostty-web` and resume live rendering
- when a terminal exits the viewport, capture latest state and release the context
- if the pool is full, lower-priority offscreen terminals stay frozen until promoted

### 5. Canvas integration

Because the canvas is rendered via `@dfosco/tiny-canvas`, virtualization should be tied to actual viewport visibility rather than just widget existence in canvas state. Dragging, zooming, panning, and expanded views should all feed the same visibility/priority model.

## Impact

This enables effectively unlimited terminal widgets per canvas while keeping the small set of visible or active terminals on full `ghostty-web` WebGL rendering.

Benefits:

- avoids browser WebGL context exhaustion
- preserves responsive interaction for active terminals
- scales to dense multi-terminal canvases
- reuses existing snapshot infrastructure instead of inventing a new persistence path

This should be treated as a future `storyboard-core` rendering improvement, not a desktop-app-only workaround.
